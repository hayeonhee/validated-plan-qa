# validated-plan-qa

validated-plan이 "다 잘 됐어"라고 말할 때, 진짜 잘 된 건지 다시 확인하는 스킬.

## 왜 필요한가

validated-plan은 자체적으로 검증을 한다. 계획을 세우고, 기준을 만들고, 실행하고, 결과를 평가한다.

문제는 이 모든 과정이 **같은 맥락 안에서** 일어난다는 것이다. 시험 문제를 낸 사람이 직접 채점하는 것과 비슷하다. 자기가 만든 기준으로 자기가 판단하니까, 기준을 슬쩍 낮추거나 애매한 걸 "됐다"고 넘기기 쉽다.

이 스킬은 그걸 잡는다.

## 어떻게 작동하는가

### 1단계: 구조가 맞는지 본다 (Layer 1)

스크립트와 에이전트가 산출물을 검사한다.

- 파일이 다 있는가
- 앞 단계의 지적이 다음 단계에 반영됐는가
- 기준이 중간에 느슨해지지 않았는가
- gap 보완이라면서 실제로는 해석만 바꾼 건 아닌가

### 2단계: 체인을 우회해서 다시 본다 (Layer 2)

별도 에이전트에게 두 가지만 준다:
- clarify-result.md (사용자 의도를 정리한 문서)
- 최종 결과물 (실제로 만들어진 것)

중간 과정은 일절 보여주지 않는다. 계획서, 검증 기준, 실행 리뷰 전부 차단한다. 그래야 계획-검증-실행 체인(Step 1~10)에서 생긴 프레이밍 오염, 기준 약화, 앵커링에 오염되지 않고 판단할 수 있다.

Step 0의 오염까지 잡기 위해, clarify-result.md(체인의 해석)가 아니라 user-request-raw.md(사용자가 처음 입력한 원문)를 "원래 의도"로 사용한다. validated-plan이 clarify 시작 전에 원문을 별도 보존하고, 독립 판정은 이 원문과 최종 결과물만 비교한다.

### 3단계: 둘을 비교한다 (Layer 3)

체인이 내린 판정과 독립 판정을 나란히 놓고 본다.

- 둘 다 PASS면: 신뢰도 높음
- 체인만 PASS이고 독립은 FAIL이면: **체인이 놓친 것이 있다** (가장 위험한 경우)
- 체인은 FAIL인데 독립은 PASS면: 기준이 과하게 엄격했을 수 있음

### 4단계: 결과를 보존한다

QA 결과를 아카이브에 저장해둔다. 나중에 "그때 QA가 맞았나?" 역추적할 때 쓴다.

## 놓친 걸 학습한다

여기까지가 "한 번의 QA에서 할 수 있는 최선"이다. 그래도 놓치는 건 있다.

그래서 사후 학습 루프가 있다:

1. **기록**: 나중에 문제가 발견되면 `missed-patterns.md`에 적는다
2. **축적**: 비슷한 문제가 반복되면 패턴이 보인다
3. **승격**: 패턴이 확인되면 체크리스트나 스크립트에 규칙으로 추가한다
4. **효과**: 다음 QA부터 그 유형의 문제를 잡을 수 있다

### 언제 기록하는가

타이머가 아니라 이벤트 기반이다. 정해진 주기로 체크하는 게 아니라, 놓친 게 발견되는 순간 기록한다.

- 사용자가 "이거 QA 통과했는데 문제 있다"고 말할 때
- 다음 작업 하다가 이전 산출물에서 결함을 발견했을 때
- PR 리뷰에서 누군가 지적했을 때
- 프로덕션에서 버그가 터졌을 때

5분 후일 수도 있고 2주 후일 수도 있다. 중요한 건 시점이 아니라 **"발견되면 즉시 적는다"**는 것이다.

### 기록할 때 가장 중요한 것

"왜 못 잡았는지"를 3가지로 분류한다:

- **규칙이 없었다** (coverage gap): 이 유형의 문제를 체크하는 항목 자체가 없었음
- **규칙은 있었는데 적용 안 됐다** (execution gap): 체크리스트에 있지만 에이전트가 건너뜀
- **적용했는데 판정이 틀렸다** (judgment gap): 확인은 했는데 "괜찮다"고 오판함

이 분류가 나중에 어디를 고쳐야 하는지 알려준다. 규칙이 없었으면 규칙을 추가하고, 적용이 안 됐으면 실행 절차를 고치고, 판정이 틀렸으면 판정 기준을 정교하게 만든다.

### 5개 모이기 전에는 어떻게 하는가

기록이 5개 미만이어도 할 수 있는 것과 못 하는 것이 있다:

**언제든 가능:**
- 발견 즉시 missed-patterns.md에 기록
- 사람이 기록을 읽고 직접 판단
- 명확한 패턴이 보이면 수동으로 규칙 추가 (승격 기준 충족 시)

**5개 이상 쌓여야 가능:**
- pattern-analyst 에이전트가 자동으로 패턴 분석
- regression-verifier 에이전트가 새 규칙 검증

5개는 **에이전트를 돌릴 최소 데이터량**이다. 데이터가 적으면 에이전트가 1-2개 사례에 과적합된 규칙을 만들 위험이 크다. 하지만 사람이 보기에 3개만으로도 패턴이 명확하면 수동 승격은 가능하다.

요약하면: 기록은 항상 한다. 자동 분석만 5개부터 돌린다.

### 규칙으로 승격하려면

5개 기준과 별개로, 규칙이 되려면 아래 5가지를 **전부** 충족해야 한다:

1. **재현 가능**: 같은 유형이 2번 이상 발생했다
2. **범용 적용**: 이 프로젝트에서만 해당하는 게 아니다
3. **규칙화 가능**: 체크리스트 항목이나 스크립트 검사로 표현할 수 있다
4. **오탐 낮음**: 정상인 걸 문제라고 잡을 위험이 낮다
5. **실제 사례 존재**: missed-patterns.md에 구체적 기록이 있다

하나라도 안 되면 승격하지 않는다. 예를 들어 2번 발생했지만 특정 프로젝트에서만 해당하면 (범용성 부족) 승격하지 않는다.

### 놓친 쪽이 규칙을 만들지 않는다

이건 의도적인 설계다.

문제를 놓친 에이전트나 세션이 "다음엔 이걸 체크해야지"라고 직접 규칙을 만들면 자기 정당화가 섞인다. "내가 놓친 건 이것뿐이야"라고 범위를 좁게 잡거나, 자기 실수를 합리화하는 방향으로 규칙을 만들 수 있다.

그래서 역할을 분리한다:
- 놓친 걸 **발견하고 기록하는 쪽** (intake)
- 패턴을 **분석하고 규칙을 제안하는 쪽** (pattern-analyst)
- 규칙이 **진짜 좋은 규칙인지 검증하는 쪽** (regression-verifier)

이 셋은 서로 다른 에이전트/세션이 담당한다.

### 규칙은 어디에 들어가는가

스킬 본문(SKILL.md)을 직접 고치는 건 최후 수단이다. 먼저 아래 순서로 검토한다:

1. 스크립트로 자동 검사할 수 있는가 -> `scripts/` 수정
2. 에이전트가 체크할 항목인가 -> `references/checklist.md` 추가
3. 새로운 실패 유형인가 -> `references/failure-taxonomy.md` 추가
4. 판정 로직 자체의 문제인가 -> `references/verdict-rules.md` 수정
5. 위 어디에도 안 맞을 때만 -> `SKILL.md` 수정

이유: SKILL.md는 실행 절차 전체를 담고 있어서 자주 고치면 복잡해진다. 참조 문서와 스크립트를 강화하는 게 더 안전하다.

## 파일 구조

```
SKILL.md                          스킬 본문 (실행 절차)
README.md                         이 문서
scripts/
  audit_validated_plan_run.py     자동 구조 검사 스크립트
references/
  checklist.md                    QA 체크리스트
  failure-taxonomy.md             실패 유형 분류
  verdict-rules.md                판정 규칙
  missed-patterns.md              놓친 문제 기록소
  promotion-rules.md              규칙 승격 기준
```

실행 시 생성되는 파일:

```
.omc/qa-archive/{task-slug}/      QA 결과 아카이브
  final-qa-report.md              최종 보고서
  independent-verdict.md          독립 판정
  layer1-findings.md              구조 검사 결과
  metadata.md                     요약 정보
```

## 사용법

validated-plan 실행이 끝난 뒤:

```
"validated-plan QA 해줘"
"실행 끝난 거 QA"
```

## 현재 상태

- 3-Layer QA 엔진: 작동 중
- QA Archive 규칙: 정의 완료, 실행 데이터 0건
- missed-patterns.md: 포맷 완료, 엔트리 0건
- promotion-rules.md: 기준 + 임계값 정의 완료
- pattern-analyst: 미구현 (엔트리 5개 이상 시 추가)
- regression-verifier: 미구현 (규칙 제안 + 아카이브 2개 이상 시 추가)
